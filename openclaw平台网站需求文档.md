# openclaw 平台网站需求文档

版本：v0.1  
日期：2026-05-28  
定位：企业内部署优先，未来预留云平台能力

## 1. 产品概述

openclaw 是面向企业内部署的智能运营平台。平台通过统一模型网关、聊天软件接入、三方系统连接器、账号匹配、SKILL 管理、token 权限管理和 Agent 编排能力，把企业内部的人、组织、系统、消息和自动化任务连接起来。

第一版优先满足企业内部真实使用场景，部署形态以 Kubernetes 为主，类似传统 ERP 在企业内部署。系统设计需要预留未来云平台和多租户能力，但第一版不以互联网 SaaS 作为主要目标。

### 1.1 核心目标

1. 统一接入多种模型 API，形成可治理、可审计、可计费的模型能力入口。
2. 把企业微信等聊天软件抽象成统一账号、消息、群、机器人和事件模型。
3. 对接 CRM、ERP、售后、设备、工单等三方系统，形成可被 Agent 和 SKILL 调用的企业能力层。
4. 建立聊天账号与三方系统账号之间的可解释匹配、人工覆盖和持续刷新机制。
5. 支持个人和组织级 Agent 流，完成定时任务、事件触发、消息通知、三方系统联动和自动巡检。
6. 管理个人或组织申请的 token，并精确控制到三方系统、账号、接口、动作、额度和有效期。
7. 提供 SKILL 管理和内部市场，支持低风险内容自由分享，高风险能力审核后发布。
8. 统计个人、组织、Agent、模型源、SKILL 的次数、token、费用和运行质量。

### 1.2 第一版原则

1. 企业内部署优先，云平台能力预留。
2. 组织级流程是主线，个人 Agent 是重要协作入口。
3. 可解释优先于黑盒自动化，尤其是模型策略、账号匹配和权限判断。
4. 所有关键动作必须有权限、审计、回滚或人工覆盖路径。
5. Kubernetes 是第一版运行环境。
6. Python 包和 Agent 运行时使用统一基础镜像，不开放个人自定义依赖。
7. token 属于个人或组织，不属于 SKILL 本身；SKILL 和 Agent 运行时只能按授权主体使用 token。

### 1.3 第一版不做

1. 不做完整互联网 SaaS 租户计费平台。
2. 不开放每个 Agent 私自安装 Python 依赖。
3. 不默认使用黑盒智能模型调度替代可解释路由策略。
4. 不允许未经审核的代码类、高权限类 SKILL 直接在组织范围发布。
5. 不把自动账号匹配结果视为不可修改的最终结果。

## 2. 典型业务场景

### 2.1 CRM 设备销售通知

销售系统中某特定型号设备售出后，CRM 触发事件进入 openclaw。平台根据设备型号、客户、销售负责人、组织规则和通知策略，调用组织级 Agent 流判断是否需要通知。通知可以转发到企业微信群、某个销售负责人、售后负责人或业务主管。

关键能力：

1. CRM 连接器接收销售事件。
2. 账号匹配模块把 CRM 用户、客户经理和聊天软件账号关联起来。
3. Agent 流按组织配置执行条件判断。
4. 通知动作可以是普通消息，也可以是一个 SKILL。
5. 调用 CRM 或企业微信接口时使用组织或个人授权 token。
6. 所有触发、判断、调用和通知结果可审计。

### 2.2 售后设备巡检

售后部门配置定时 Agent，每天或每小时检查重点设备状态、工单状态、告警状态和客户反馈。如果发现异常，Agent 调用模型进行摘要和判断，再通过企业微信通知对应负责人。

关键能力：

1. 定时触发。
2. ERP、CRM、工单、设备平台等三方系统查询。
3. 模型策略按任务类型选择模型源。
4. Agent 使用组织级 SKILL 和组织 token。
5. 异常通知支持升级、重试和人工确认。

### 2.3 摄像头图像巡检

组织配置 Agent 每隔几秒或几分钟拉取摄像头图片，检查镜头是否脏污、遮挡、偏移或离线。模型源可使用视觉模型 API。异常结果通过 IM 推送，也可以写回工单系统。

关键能力：

1. 高并发定时任务和任务限流。
2. 视觉模型源接入。
3. 按摄像头、区域、负责人和组织权限分发通知。
4. 图像数据和分析结果需要按配额和保留策略管理。
5. 对误报、漏报、人工确认结果做记录，未来用于调度和提示优化。

### 2.4 个人与组织协同 Agent

员工可以拥有个人 Agent，用于个人消息整理、系统查询和待办提醒。组织可以拥有组织级 Agent，用于部门流程、统一通知、系统巡检和业务自动化。组织 Agent 可以把消息转发给个人，个人也可以触发组织授权范围内的流程。

关键能力：

1. 个人和组织都是授权主体。
2. Agent 流可配置、可编辑、可测试、可启停。
3. Agent 可以使用个人或组织的 SKILL 和 token，但必须符合权限边界。
4. 组织到个人、个人到组织的 IM 转发需要审计。

## 3. 平台分层

### 3.1 业务场景层

承载销售通知、售后巡检、图像巡检、组织自动任务、个人助手、业务摘要、告警分发等具体场景。

### 3.2 交互入口层

包括管理后台、聊天软件适配器、三方系统连接器、Webhook、事件订阅和 API 入口。

第一版聊天软件以企业微信适配器作为验收样例，但底层需要抽象聊天软件模型，预留钉钉、飞书、Slack 等适配器。

### 3.3 编排与身份层

包括 Agent 流、触发器、条件、动作、账号匹配、组织架构同步、个人与组织授权主体。该层决定“谁在什么条件下，用什么权限，执行什么动作”。

### 3.4 能力与资源层

包括模型源、模型策略网关、SKILL 管理、SKILL 市场、token 管理、用户磁盘配额。

### 3.5 治理与运维层

包括 Kubernetes 状态管理、统一基础镜像、服务重启、健康检查、运行日志、审计、用量统计、费用归集和平台配置。

## 4. 核心对象模型

### 4.1 用户

用户代表企业内部自然人。用户可以绑定多个聊天软件账号、多个三方系统账号、多个邮箱、多个手机号。用户可以拥有个人 Agent、个人 token、个人 SKILL 可见范围和个人磁盘配额。

关键字段：

1. 用户 ID。
2. 姓名。
3. 主邮箱。
4. 邮箱列表。
5. 手机号列表。
6. 所属组织。
7. 岗位或角色。
8. 状态：启用、停用、离职、待确认。

### 4.2 组织

组织代表部门、团队、项目组、岗位组或从三方系统同步来的组织节点。组织可以拥有 Agent、token、SKILL 可见范围、用量预算和策略配置。

关键字段：

1. 组织 ID。
2. 组织名称。
3. 父组织。
4. 组织类型。
5. 成员列表。
6. 管理员列表。
7. 状态。

### 4.3 聊天账号

聊天账号是用户在企业微信、钉钉、飞书等聊天软件中的身份。平台不把企业微信账号直接等同于用户，而是通过账号匹配模块建立关系。

关键字段：

1. 聊天软件类型。
2. 外部账号 ID。
3. 昵称。
4. 手机号或邮箱。
5. 所属聊天组织。
6. 绑定用户。
7. 匹配状态。

### 4.4 三方系统账号

三方系统账号是用户在 CRM、ERP、工单、设备平台等系统中的身份。

关键字段：

1. 系统类型。
2. 外部账号 ID。
3. 登录名。
4. 姓名。
5. 邮箱。
6. 手机号。
7. 部门。
8. 绑定用户。
9. 匹配状态。

### 4.5 token

token 是个人或组织申请、持有和被管理的授权资源。token 可以代表三方系统账号、API 密钥、OAuth 授权、服务账号或内部接口访问凭证。

token 不属于 SKILL。SKILL 和 Agent 运行时必须声明使用哪个个人或组织授权主体，并在权限范围内调用。

关键字段：

1. token ID。
2. 归属主体：个人或组织。
3. 目标系统。
4. 授权账号。
5. 允许接口。
6. 允许动作。
7. 额度限制。
8. 有效期。
9. 审批状态。
10. 最近使用时间。
11. 风险等级。

### 4.6 SKILL

SKILL 是平台可复用能力单元，可以是提示词模板、操作流程、工具说明、内部能力封装或高级代码插件。第一版采用分级治理。

普通 SKILL：

1. 不包含代码执行。
2. 不直接要求 token。
3. 主要是提示词、流程说明、表单配置或内部无风险模块。
4. 可由员工上传分享，并设置可见范围。

高级 SKILL：

1. 涉及代码执行、外部接口、token 使用、组织权限或高风险动作。
2. 必须审核后发布。
3. 必须声明依赖、权限、输入输出、风险等级和审计要求。

### 4.7 Agent

Agent 是可配置的自动化执行单元。Agent 可以归属于个人或组织，可以定时执行、事件触发、手动触发，也可以通过聊天软件触发。

关键字段：

1. Agent ID。
2. 名称。
3. 归属主体：个人或组织。
4. 触发器。
5. 条件。
6. 动作。
7. 使用的 SKILL。
8. 使用的 token 授权主体。
9. 模型策略。
10. 运行状态。
11. 最近运行记录。
12. 失败处理策略。

### 4.8 模型源

模型源代表可调用的模型 API 或模型服务。可以是外部模型供应商、私有化模型服务、视觉模型、语音模型、文本模型或嵌入模型。

关键字段：

1. 模型源 ID。
2. 供应商。
3. 模型名称。
4. API 地址。
5. 鉴权方式。
6. 上下文长度。
7. 能力标签。
8. 单价。
9. 限流配置。
10. 健康状态。

### 4.9 模型策略

模型策略定义如何使用模型源。第一版按可解释模型网关设计，类似 nginx 的上游池、权重、轮转和故障摘除。

关键字段：

1. 策略 ID。
2. 适用组织、用户、Agent 或任务类型。
3. 上游模型池。
4. 权重。
5. 轮转方式。
6. 超时。
7. 重试。
8. 熔断。
9. 降级。
10. 成本上限。
11. 日志等级。

## 5. 模块需求

### 5.1 模型源模块

#### 5.1.1 目标

统一管理各种模型 API 接入，为平台内 Agent、SKILL、聊天问答和后台任务提供稳定模型能力。

#### 5.1.2 功能需求

1. 支持新增、编辑、停用模型源。
2. 支持文本、视觉、语音、嵌入等模型能力标签。
3. 支持 API 地址、鉴权方式、模型名称、上下文长度、单价、限流配置。
4. 支持模型源健康检查。
5. 支持测试调用。
6. 支持调用日志和错误日志。
7. 支持按组织、用户、Agent、SKILL 限制可用模型源。

#### 5.1.3 异常处理

1. API 超时需要记录超时时间、模型源、调用主体和请求类型。
2. 鉴权失败需要自动标记风险状态，并提醒管理员。
3. 连续失败达到阈值时模型源进入异常状态，由模型策略决定是否摘除。

#### 5.1.4 验收标准

1. 管理员可以配置至少两个模型源。
2. Agent 可以通过模型策略调用模型源。
3. 模型源失败时有日志、状态和告警。
4. 用量统计可以区分不同模型源的调用次数、token 和费用。

### 5.2 模型策略模块

#### 5.2.1 目标

构建可解释的模型网关，支持类似 nginx 的上游池、权重、轮转、故障摘除、重试、超时、熔断、降级和灰度。

#### 5.2.2 功能需求

1. 支持创建模型上游池。
2. 支持按权重轮转模型源。
3. 支持按任务类型路由，例如聊天、摘要、视觉识别、结构化抽取。
4. 支持按组织、用户、Agent 绑定默认策略。
5. 支持超时、重试、熔断、降级配置。
6. 支持灰度策略，例如某组织 10% 请求使用新模型。
7. 支持成本上限和预算提醒。
8. 支持策略模拟测试。

#### 5.2.3 演进能力

未来可增加基于成本、延迟、上下文长度、历史质量和人工反馈的智能调度。但第一版不把智能调度作为核心验收目标。

#### 5.2.4 验收标准

1. 同一任务可以按策略路由到不同模型源。
2. 某模型源失败后，策略能自动摘除或降级。
3. 管理员能看到每次请求命中了哪个策略和哪个模型源。

### 5.3 聊天软件对接模块

#### 5.3.1 目标

抽象聊天软件能力，第一版以企业微信适配器为验收样例，未来预留钉钉、飞书、Slack 等适配器。

#### 5.3.2 统一模型

1. 聊天软件。
2. 聊天账号。
3. 群。
4. 机器人。
5. 消息。
6. 事件。
7. 回调。
8. 附件。

#### 5.3.3 功能需求

1. 支持聊天账号同步。
2. 支持组织和部门同步。
3. 支持单聊消息发送。
4. 支持群消息发送。
5. 支持机器人回调。
6. 支持消息事件触发 Agent。
7. 支持文件、图片、链接等基础消息类型。
8. 支持发送失败重试和人工补发。

#### 5.3.4 验收标准

1. 企业微信账号可同步到平台。
2. 企业微信消息可触发 Agent。
3. Agent 可向企业微信个人或群发送消息。
4. 聊天账号不直接等同于平台用户，必须经过账号匹配模块绑定。

### 5.4 三方系统对接模块

#### 5.4.1 目标

把 CRM、ERP、工单、售后、设备、财务等三方系统抽象为统一连接器，供 Agent、SKILL 和后台功能调用。

#### 5.4.2 功能需求

1. 支持连接器注册、启用、停用。
2. 支持 API 地址、鉴权方式、接口清单、字段映射。
3. 支持 Webhook、轮询、手动同步三种事件来源。
4. 支持接口级权限声明。
5. 支持连接测试。
6. 支持数据同步日志。
7. 支持接口调用审计。

#### 5.4.3 典型系统

1. CRM：客户、商机、合同、设备销售、联系人。
2. ERP：订单、库存、发货、财务状态。
3. 工单系统：工单、处理人、状态、时限。
4. 设备平台：设备状态、告警、图像、传感器数据。

#### 5.4.4 验收标准

1. 至少支持一个 CRM 或 ERP 连接器完成账号同步和业务事件触发。
2. 连接器接口能被 token 权限模块精确管控。
3. Agent 调用三方接口时能记录接口、主体、参数摘要和结果。

### 5.5 账号匹配模块

#### 5.5.1 目标

把聊天系统账号和三方系统账号匹配到平台用户。匹配结果必须可解释、可人工覆盖、可持续刷新。

#### 5.5.2 匹配字段

1. 主邮箱。
2. 邮箱别名。
3. 手机号。
4. 企业微信账号。
5. CRM 或 ERP 用户号。
6. 姓名。
7. 部门。
8. 岗位。
9. 外部系统唯一 ID。

#### 5.5.3 匹配策略

1. 默认保守策略：强一致字段才自动绑定。
2. 支持每个三方系统配置字段、权重、阈值。
3. 高分自动匹配。
4. 中分进入人工确认。
5. 冲突进入冲突队列。
6. 无法匹配进入未匹配队列。

#### 5.5.4 人工覆盖

1. 自动匹配成功后，管理员仍可重新人工覆盖。
2. 自动匹配失败后，管理员可手工绑定。
3. 冲突项可由管理员选择正确用户，也可标记为暂不处理。
4. 人工覆盖需要记录操作人、原因、时间、来源和有效期。
5. 后续自动刷新不得静默覆盖人工结果，除非管理员配置允许。

#### 5.5.5 持续刷新

1. 支持定时重新同步外部账号。
2. 支持检测邮箱、手机号、部门变化。
3. 支持发现新增账号、离职账号、重复账号。
4. 支持匹配结果变更提醒。

#### 5.5.6 验收标准

1. 平台能展示匹配成功、待确认、冲突、未匹配四类列表。
2. 任意匹配结果都能看到匹配依据。
3. 管理员可人工覆盖成功和失败的匹配结果。
4. 人工覆盖结果有审计记录。

### 5.6 SKILL 管理模块

#### 5.6.1 目标

管理个人、组织、岗位或三方组织架构对应的 SKILL 可见范围和使用权限。

#### 5.6.2 功能需求

1. 支持创建普通 SKILL。
2. 支持创建高级 SKILL。
3. 支持按个人、组织、岗位、角色设置可见范围。
4. 支持与三方组织架构同步。
5. 支持 SKILL 版本管理。
6. 支持启用、停用、下架。
7. 支持输入输出说明。
8. 支持风险等级。
9. 支持使用记录。

#### 5.6.3 权限规则

1. 用户只能看到自己或所属组织可见的 SKILL。
2. 组织 Agent 可以使用组织授权范围内的 SKILL。
3. 个人 Agent 可以使用个人授权范围内的 SKILL。
4. 高级 SKILL 使用 token 或接口时，必须通过 token 权限模块校验。

#### 5.6.4 验收标准

1. 管理员可以给某组织配置可见 SKILL。
2. 用户只能看到自己可见范围内的 SKILL。
3. Agent 调用 SKILL 时能校验权限并记录日志。

### 5.7 token 权限管理模块

#### 5.7.1 目标

管理个人和组织申请的 token，并将权限精确控制到三方系统、账号、接口、动作、额度和有效期。

#### 5.7.2 功能需求

1. 支持个人申请 token。
2. 支持组织申请 token。
3. 支持管理员审批、拒绝、撤销。
4. 支持接口级权限配置。
5. 支持动作级权限配置，例如读取、创建、更新、删除、发送消息。
6. 支持额度限制。
7. 支持有效期。
8. 支持到期提醒。
9. 支持异常调用告警。
10. 支持 token 使用审计。

#### 5.7.3 运行时规则

1. Agent 或 SKILL 调用接口前，必须声明授权主体。
2. 授权主体可以是个人，也可以是组织。
3. 平台校验主体是否拥有目标系统、目标账号、目标接口和目标动作权限。
4. 校验通过后才允许调用。
5. 调用结果必须记录。

#### 5.7.4 验收标准

1. 某用户可以申请 CRM 查询接口 token。
2. 某销售组织可以申请 CRM 事件和企业微信通知 token。
3. Agent 调用未授权接口时必须失败，并返回明确原因。
4. 管理员可以撤销 token，撤销后相关 Agent 调用失败。

### 5.8 磁盘管理模块

#### 5.8.1 目标

在 Kubernetes 部署环境下，管理每个人的 Agent 工作区空间配额、用量观察、清理和调整。

#### 5.8.2 功能需求

1. 支持设置用户磁盘配额。
2. 支持查看用户已用空间。
3. 支持按 Agent、文件类型、时间查看空间占用。
4. 支持配额告警。
5. 支持管理员调整配额。
6. 支持用户申请扩容。
7. 支持按策略清理临时文件、缓存、历史运行产物。

#### 5.8.3 不做范围

第一版不把磁盘管理扩展为复杂存储平台，只管理用户工作区配额和可观测性。

#### 5.8.4 验收标准

1. 管理员能查看每个用户的配额和用量。
2. 用户超过配额时 Agent 新增写入会被限制或进入失败状态。
3. 管理员可调整用户配额。

### 5.9 状态管理模块

#### 5.9.1 目标

在 Kubernetes 上管理 openclaw 中心服务、Agent 运行状态、统一基础镜像、运行日志、健康检查和必要的重启操作。

#### 5.9.2 功能需求

1. 展示平台核心服务状态。
2. 展示 Agent 运行状态。
3. 支持管理员重启 openclaw 中心服务。
4. 支持管理员重启失败或卡住的 Agent 运行实例。
5. 展示 Kubernetes Pod 状态摘要。
6. 展示统一基础镜像版本。
7. 展示平台 Python 包版本。
8. 支持发布平台基础镜像升级记录。
9. 支持任务队列状态和失败任务查看。

#### 5.9.3 运行时原则

1. 第一版使用统一基础镜像。
2. 所有人运行环境一致。
3. 不允许个人 Agent 私自安装依赖。
4. 如需新增依赖，走平台版本升级流程。

#### 5.9.4 验收标准

1. 管理员能看到中心服务和 Agent 的状态。
2. 管理员能执行受控重启。
3. 平台能展示当前统一基础镜像和包版本。

### 5.10 Agent 模块

#### 5.10.1 目标

支持个人和组织拥有 Agent 流。Agent 可以定时执行、事件触发、手动测试、编辑配置，并使用个人或组织的 SKILL 和 token 权限。

#### 5.10.2 配置模式

采用双层模式：

1. 技术人员定义可复用 SKILL、连接器、模板、安全边界和运行参数。
2. 业务负责人基于模板配置触发器、条件、动作、通知目标和启停规则。

#### 5.10.3 触发器

1. 定时触发。
2. 手动触发。
3. 聊天消息触发。
4. 三方系统事件触发。
5. Webhook 触发。
6. 文件或数据变更触发。

#### 5.10.4 条件

1. 字段匹配。
2. 时间窗口。
3. 组织范围。
4. 设备型号。
5. 客户类型。
6. 告警等级。
7. 模型判断结果。
8. 人工审批结果。

#### 5.10.5 动作

1. 调用模型。
2. 调用 SKILL。
3. 查询三方系统。
4. 写回三方系统。
5. 发送 IM 消息。
6. 创建工单。
7. 更新 CRM 或 ERP 字段。
8. 触发另一个 Agent。
9. 等待人工确认。

#### 5.10.6 测试与发布

1. Agent 配置可保存草稿。
2. 支持测试运行。
3. 测试运行必须显示输入、命中条件、调用步骤、权限校验和输出。
4. 发布前检查 token、SKILL、模型策略和通知目标是否可用。
5. 支持启用、停用、回滚版本。

#### 5.10.7 失败处理

1. 支持重试。
2. 支持失败通知。
3. 支持进入人工处理队列。
4. 支持跳过并继续下一步。
5. 支持熔断。

#### 5.10.8 验收标准

1. 组织管理员可以配置一个 CRM 事件触发的 Agent。
2. 用户可以配置一个个人定时提醒 Agent。
3. Agent 可以测试运行并展示每一步结果。
4. Agent 可以使用个人或组织 token，但越权调用必须失败。

### 5.11 SKILL 市场模块

#### 5.11.1 目标

支持内部 SKILL 上传、分享、搜索、安装、版本管理和可见范围控制。

#### 5.11.2 上传规则

1. 普通 SKILL 可由员工上传分享。
2. 普通 SKILL 不得包含代码执行、高风险接口和 token 要求。
3. 高级 SKILL 必须审核。
4. 高级 SKILL 需要声明权限、依赖、风险、输入输出和审计要求。

#### 5.11.3 可见范围

1. 仅自己可见。
2. 指定用户可见。
3. 指定组织可见。
4. 全公司可见。
5. 管理员可强制下架或调整范围。

#### 5.11.4 功能需求

1. 支持上传。
2. 支持编辑。
3. 支持版本发布。
4. 支持搜索。
5. 支持分类。
6. 支持收藏。
7. 支持安装到个人或组织。
8. 支持评分或反馈。
9. 支持下架。

#### 5.11.5 验收标准

1. 普通员工可以上传无 token、无代码的普通 SKILL。
2. 高级 SKILL 未审核前不能被组织 Agent 使用。
3. 用户只能看到自己可见范围内的市场内容。

## 6. 权限与审计

### 6.1 权限主体

1. 用户。
2. 组织。
3. 管理员。
4. Agent。
5. SKILL。
6. 连接器。

Agent 和 SKILL 不是最终权限归属主体。它们运行时必须绑定个人或组织授权。

### 6.2 权限范围

1. 模型源可用范围。
2. 模型策略可用范围。
3. SKILL 可见和使用范围。
4. token 接口权限。
5. 三方系统数据范围。
6. 聊天消息发送范围。
7. Agent 编辑、测试、启停范围。
8. 磁盘配额管理范围。

### 6.3 审计日志

必须记录：

1. 谁触发。
2. 以哪个授权主体执行。
3. 调用了哪个 Agent。
4. 使用了哪个 SKILL。
5. 使用了哪个 token。
6. 调用了哪个模型源。
7. 调用了哪个三方接口。
8. 输入和输出摘要。
9. 成功或失败原因。
10. 时间、耗时、费用。

敏感内容不应默认展示明文。token、密钥、完整请求体和敏感业务字段需要脱敏或按权限查看。

## 9. 关键验收场景

### 9.1 设备销售通知闭环

1. CRM 产生设备销售事件。
2. openclaw 接收事件。
3. 平台匹配 CRM 账号和企业微信账号。
4. 组织 Agent 判断设备型号和通知规则。
5. Agent 使用组织 token 查询补充信息。
6. Agent 使用企业微信适配器发送消息。
7. 平台记录模型调用、接口调用、通知结果和费用。

### 9.2 账号冲突处理闭环

1. 同一手机号匹配到两个用户。
2. 系统进入冲突队列。
3. 管理员查看匹配依据。
4. 管理员人工选择正确用户。
5. 系统记录覆盖原因。
6. 后续自动刷新不得静默覆盖该结果。

### 9.3 token 越权拦截闭环

1. 某 Agent 尝试调用未授权 ERP 更新接口。
2. token 权限模块拒绝调用。
3. Agent 进入失败处理策略。
4. 管理员可查看失败原因。
5. 审计日志记录授权主体、接口和拒绝原因。

### 9.4 模型源故障降级闭环

1. 主模型源连续超时。
2. 模型策略触发熔断。
3. 请求切换到备用模型源。
4. 平台记录策略命中和故障摘除。
5. 管理员可在后台看到模型源异常状态。

## 10. 风险与待决事项

### 10.1 风险

1. 账号误匹配会导致权限和通知错误，必须默认保守并支持人工覆盖。
2. token 权限过粗会造成数据泄露或误操作，必须接口级、动作级控制。
3. Agent 自动化如果缺少测试运行和审计，会难以排查错误。
4. SKILL 市场如果不分级，容易把低风险分享和高风险代码执行混在一起。
5. 模型策略如果过早黑盒化，会导致成本、质量和故障不可解释。

### 10.2 待决事项

1. 第一版首个 CRM 或 ERP 连接器选择哪个系统。
2. 企业微信适配器需要支持的消息类型清单。
3. 摄像头图像巡检是否进入第一版正式验收，还是作为演示场景。
4. token 审批流是否需要多级审批。
5. 用量费用是否需要对接企业内部财务科目。
6. 审计日志保留周期。
